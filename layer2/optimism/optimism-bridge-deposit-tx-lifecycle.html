执行顺序

L1 事件

Portal 在 L1 上发出 TransactionDeposited，把跨链执行所需字段编码进 log。

02

派生 DepositTx

op-node 在新 epoch 首块扫描 receipts，解析事件并构造用户 DepositTx。

03

构建属性

把 L1InfoDepositTx、用户 DepositTx 和升级交易装入 PayloadAttributes。

04

L2 执行

执行引擎按顺序执行交易，最终产出 ExecutionPayload 并完成合约调用链。

Deposit Decoder

从 topics 解析 from / to / version，从 opaque data 解析 mint / value / gas / isCreation / data，得到 types.DepositTx 的核心字段。

sourceHash 确定性身份

用户 deposit 走 domain=0 规则，使用 keccak256(domain || keccak256(l1BlockHash || logIndex)) 作为唯一来源标识。

L1Info Builder

L1InfoDepositTx 永远在最前

L1InfoDeposit() 构造一笔系统 deposit tx，把 L1 block number、timestamp、basefee、batcherHash、fee scalar 等写入 L2 的 L1Block 预部署合约。

PayloadAttributes

交易顺序被固定

txs = [L1InfoDepositTx] + [UserDepositTx...] + [UpgradeTx...] (+ 普通交易，sequencer 模式下)。验证者模式通常开启 NoTxPool=true，只允许确定性派生交易进入区块。

DepositTx vs 普通交易
execution semantics
type 0x7E
维度 普通 L2 交易 DepositTx
签名 必须验签 不验签，from 由 L1 日志直接给出

ETH 存款路径

  • DepositTx 的 to 一般是 L2 上的 L2CrossDomainMessenger。
  • 先执行 relayMessage()，验证 msg.sender 逆别名后确实来自 L1 messenger。
  • 随后通过 SafeCall.call() 调到 L2StandardBridge.finalizeBridgeETH()。
  • 最终再由 bridge 通过 SafeCall.call(to, value, "") 把 ETH 转给接收者。

ERC20 存款路径

  • DepositTx 本身通常 value=0、mint=0，只负责把跨域消息带到 L2。
  • L2StandardBridge.finalizeBridgeERC20() 检查本地 token 是否为 OptimismMintableERC20。
  • 通过本地 token 的 mint(to, amount) 在 L2 铸造对应份额。
  • 因此 ERC20 的“资产落地”发生在 token 合约 mint，而不是 DepositTx 自身的 ETH mint 语义。
为什么 SafeCall 关键： SafeCall.call() 在底层使用 EVM CALL，但默认不复制返回数据， 可以避免 returndata bomb，并通过 hasMinGas 检查 EIP-150 下子调用是否还能拿到足够 gas。
