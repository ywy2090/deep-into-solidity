# 分布式系统中的 Safety 与 Liveness

> 在分布式系统与共识协议中，**Safety（安全性）** 和 **Liveness（活性）** 是两条最基础的性质。本文细化其定义、关系、权衡，并对照 OP Stack / 区块链场景说明。

---

## 1. 定义（形式化直觉）

### 1.1 Safety（安全性）

**直观含义**：「坏事永远不发生。」

- **形式化**：若系统在某个时刻**违反**了 Safety 性质，则该违反**不可撤销**——之后无论如何运行，都无法“抹掉”这次违反。
- **等价说法**：Safety 一旦被违反，就**永远**是错的；没有“后来改对”的可能。

**典型 Safety 性质举例**：

| 系统类型 | Safety 性质 | 违反即「坏事」 |
|----------|-------------|----------------|
| 共识/区块链 | **一致性（Agreement）**：诚实节点最终对同一区块/同一顺序达成一致 | 有人认为区块是 A，有人认为是 B，且无法再统一 |
| 区块链 | **链有效性**：区块内容、状态转移符合协议规则 | 双花、无效状态、非法交易被确认 |
| Rollup | **派生确定性**：L2 状态可由 L1 数据 + 规则唯一重放 | 不同节点从同一 L1 推导出不同 L2 状态 |
| 跨链/桥 | **消息来源与顺序**：只有经合法路径、按约定顺序执行的消息才被接受 | 伪造消息被执行、重放或乱序 |

**记忆要点**：Safety = 「不能错」；错了就**永久错**。

---

### 1.2 Liveness（活性）

**直观含义**：「好事最终会发生。」

- **形式化**：在**假定** Safety 不被违反的前提下，系统在有限时间内**一定会**取得进展（例如产出新区块、完成某笔交易、达成共识）。
- **等价说法**：只要系统还在按协议运行且没有先违反 Safety，那么**最终**会向前推进，不会永远卡死。

**典型 Liveness 性质举例**：

| 系统类型 | Liveness 性质 | 「好事」指什么 |
|----------|----------------|----------------|
| 共识/区块链 | **终局性（Termination）**：诚实节点最终对某一轮输出达成共识 | 本轮共识会结束，不会永远悬而未决 |
| 区块链 | **链增长**：在诚实多数下，链会持续出块 | 新区块会不断被生产并得到确认 |
| Rollup | **派生能追上**：在 L1 正常、数据可得时，L2 能持续从 L1 派生并推进 safe/finalized head | 不会永远停在某个高度不动 |
| 桥/消息传递 | **消息最终被中继**：合法消息在有限延迟内被目标链处理 | 存款/提款不会永远“在途” |

**记忆要点**：Liveness = 「最终会动」；在**不破坏 Safety** 的前提下**不会永远停**。

## 2. 关系与不可能三角

### 2.1 为什么常一起提？

- **Safety** 回答：「我们达成的结果**对不对**？」（有没有双花、有没有分叉被同时确认等）
- **Liveness** 回答：「我们**会不会**达成结果？」（会不会永远出不了块、永远达不成共识）

两者一起才能既「对」又「有进展」；只保证其一会导致要么错、要么死。

### 2.2 FLP 与 CAP 的直觉

- **FLP 不可能**：在异步网络、允许一个节点宕机的前提下，**不存在**同时满足 Safety 和 Liveness 的确定性共识协议。因此实际系统会：
  - 要么引入**同步假设**（超时、轮次）或**随机性**（例如 PoS 中的 VDF/VRF）来恢复 Liveness；
  - 要么在极端情况下**暂时牺牲 Liveness**（例如暂停出块、不确认新区块），以**绝不**违反 Safety。
- **CAP**：在分区（Partition）发生时，往往只能在 **C（一致性，偏 Safety）** 和 **A（可用性，偏 Liveness）** 之间倾向其一；区块链通常选 C，宁肯暂时不可用也不双花。

### 2.3 设计原则（一句话）

> **Safety 不可妥协；Liveness 可在极端情况下暂时牺牲，并通过协议与运维恢复。**

## 3. 在区块链 / Rollup 中的对应

### 3.1 链头层次：Unsafe / Safe / Finalized

| 层次 | 含义 | 偏 Safety 还是 Liveness |
|------|------|-------------------------|
| **Unsafe head** | 刚产出或刚收到、尚未被 L1/共识充分验证的区块 | 追求**进展**（Liveness）：先有块，再验证 |
| **Safe head** | 已可由 L1 数据 + 规则**确定性重放**的 L2 区块，无分歧 | **Safety**：大家一致认同的“已确认”前缀 |
| **Finalized head** | 已在 L1 上终局（或经争议期/欺诈证明无异议）的区块 | **Safety 更强**：不可回滚的终局 |

- 把区块从 unsafe 推到 safe/finalized，就是在**不破坏 Safety** 的前提下，把“可能错”的进展变成“大家一致认同”的进展。
- 若出现 L1 重组或异常，节点会**回滚 unsafe/safe**、甚至暂停派生，以**保住 Safety**，暂时牺牲一段时间的 Liveness（不出新块或不再推进 safe head）。

### 3.2 OP Stack 中的具体体现

| 机制 | Safety 作用 | Liveness 作用 |
|------|-------------|----------------|
| **Derivation（派生）** | 只接受符合规则的 L1 数据与 batch，保证 L2 状态与 L1 一致、可重放 | 持续派生使 safe head 向前推进，链不会永远停 |
| **Safe Lag 检查** | 当 unsafe 领先 safe 过多时**暂停出块**，避免在“未验证前缀”上堆太多块，降低分叉与不一致风险 | 正常时允许一定 lag，保证出块不因单次延迟就停 |
| **争议期 / 欺诈证明** | 错误断言可在争议期内被挑战，最终只有正确状态被接受 → **Safety** | 争议期结束即 finalize → **Liveness**（终局性） |
| **Sequencer 暂停 / 恢复模式** | 异常时只出空块或停出块，避免错误交易被确认 | 恢复后继续出块，保证长期 Liveness |

## 4. 细化：什么算「违反 Safety」？

在 Rollup / 桥语境下，典型 Safety 违反包括：

1. **双花**：同一笔钱在 L1 或 L2 被确认花两次。
2. **无效状态转移**：某区块的状态根或交易执行结果不符合协议规则，但仍被当作有效接受。
3. **派生不一致**：两个诚实节点从同一 L1 历史得到不同的 L2 状态或区块序列。
4. **消息伪造/重放**：未从合法 L1 路径来的“存款”或“提款”被 L2 执行，或同一笔跨链消息被重复执行。
5. **终局回滚**：已被视为 finalized 的区块在协议规则下被回滚（除非协议明确允许某种回滚且受控）。

上述任一条发生，即视为 Safety 被破坏；且一旦发生，无法通过“再跑一会儿”自动修复，只能通过社会/治理或硬分叉处理。

---

## 5. 细化：什么算「丧失 Liveness」？

在 Rollup / 桥语境下，典型 Liveness 丧失包括：

1. **共识卡死**：共识层永远无法对下一高度达成一致，链高不再增长。
2. **Sequencer 长期不出块**：排序器故障、Safe Lag 一直不满足、或配置错误导致长期不生产新区块。
3. **派生停滞**：L1 数据可得且正确，但 op-node 因 bug 或配置无法将 safe head 向前推。
4. **桥消息永远不中继**：合法提款/存款在协议与数据都正常的情况下，永远未被目标链处理。

注意：**暂时**的延迟或暂停（例如几分钟不出块、争议期内未 finalize）不必然算“丧失 Liveness”，只有**无限期**或**无上界**的停滞才算。工程上常用**超时 + 重试 / 告警 / 运维**来避免“永远停”。

## 6. 小结对照表

| 维度 | Safety（安全性） | Liveness（活性） |
|------|------------------|------------------|
| **核心问题** | 结果**对不对**？ | 结果**会不会发生**？ |
| **违反/丧失的后果** | 永久性错误（双花、分叉、无效状态等） | 系统停滞、无法取得进展 |
| **设计优先级** | 绝对优先；宁停不错 | 在保证 Safety 前提下尽量保证 |
| **典型手段** | 验证、一致性协议、争议期、回滚/暂停 | 超时、重试、恢复模式、终局规则 |
| **在 OP 中的体现** | Safe/Finalized 头、派生确定性、Safe Lag 限速、争议游戏 | 持续出块、派生推进、争议期结束即终局 |
